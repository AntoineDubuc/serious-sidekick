/serious-research

Runs a structured research operation. Mode is auto-detected based on your request, or force it with --quick or --deep.

Quick Mode

Focused Investigation

  • Single-threaded research
  • Persona reviews
  • Markdown deliverable
Bug diagnosis, focused investigation, clear single-angle questions
Deep Mode

Comprehensive Analysis

  • Parallel thread agents
  • Evidence grading (A-F)
  • Adversarial verification
  • QA citation checking
  • HTML report
Architecture decisions, competitive analysis, multi-dimensional topics

During scoping, you choose a research scope — online only, codebase only, or both — so agents only use the tools that matter (no pointless web searches for a local bug hunt, no codebase exploration for a tech evaluation).

A Stop hook uses the .active-research breadcrumb to track notebook.md across tool calls.

Research output is organized by type:

Research/ ├── bugs/ │ └── {slug}/ # notebook.md, research.md, evidence, report.html ├── features/ │ └── {slug}/ └── exploratory/ └── {slug}/

/serious-plan

Generates a v6 implementation plan from research, a PRD, or even a verbal description. Auto-detects existing /serious-research output or asks what input you have. Supports single or multiple plans with a phase map for parallel execution.

  • TDD protocol — RED → GREEN → VERIFY per acceptance criterion
  • Adaptive Persona Pipeline — severity-weighted convergence: Critical / Major / Minor. Any critical → re-review, 3+ majors → re-review, minors only → done, max 3 rounds
  • Inline QA Protocol v6 — every criterion gets independent QA sub-agent verification
  • Multi-plan support — single plan (3-7 tasks) or multiple plans with phase_map.md for parallel execution
  • Cross-plan integration review — Integration Architect + Merge Strategist (+ optional Concurrency Engineer). Same convergence rules, max 3 integration rounds. If fixes change a plan → one more individual review round
  • Evidence reports — markdown evidence generated per task

/serious-code

Executes implementation plans produced by /serious-plan. Uses a 3-level agent architecture: Orchestrator → Plan Agents → Agent Teams.

The Orchestrator manages phases and user approval. Plan Agents (one per plan per phase) execute via git worktrees for isolation. Agent Teams (5 agents per task) handle TDD implementation and verification.

  • Multi-plan execution — reads phase_map.md and runs plans in parallel phases via git worktrees
  • 5 Agent Teams agents — implementer (TDD), code reviewer, test runner, runtime checker, QA spot-checker
  • Phase-by-phase approval — user approves each phase before execution begins
  • Two-level tracking — execution_log.md (orchestrator level) + per-plan progress.md
  • Evidence generation — every task produces implementation, review, test, runtime, and QA reports
  • Failure handling — failed plans stop immediately; other parallel plans continue; user decides to fix, skip, rollback, or abort
  • Resume mode — reads execution_log.md + progress.md to find the stopping point and pick up where it left off
  • Completion report — generated at end of execution with full evidence summary
  • Stop hook tracking — uses .active-code breadcrumb to track execution_log.md

/serious-review

Structured review and defect capture that bridges /serious-code output back into the pipeline. Not a testing framework — a human review phase that collects feedback and cycles it through /serious-research → /serious-plan → /serious-code.

The Flow

  • Auto-detect context — scans for recent /serious-code output, flags missing deliverables
  • Confirm scope — presents what was found, asks to confirm
  • Choose mode — live capture, QA plan first, or both (default)
  • Live capture — each issue gets an ID (REVIEW-001), classified, severity-rated, and written to disk immediately
  • Synthesize — consolidates findings into review-summary.md formatted as research input
  • Hand off — offers to kick off the pipeline again on the findings

Issue Classification

Types

What gets captured

  • 🐛 Bug — broken behavior
  • ⚠️ Missed requirement
  • 💡 Improvement
  • 🔒 Security concern
  • 🎨 UX/UI issue
Severities

Priority levels

  • Critical — blocks usage, data loss
  • High — major feature broken
  • Medium — works but wrong
  • Low — minor polish

Output is organized in a QA folder:

QA/ └── {slug}/ ├── findings.md # All issues with IDs, types, severities ├── qa-plan.md # Generated QA checklist (if requested) └── review-summary.md # Final synthesis — feeds into /serious-research

/serious-bananas

Generate images (especially diagrams) using Google's Gemini native image generation API (Nano Banana). Interviews you with 6 questions, crafts an optimized prompt, and runs the API call.

Interview questions

  • What do you need? — plain language description of the image
  • What type? — flowchart, architecture, sequence, ER, infographic, concept map, freeform
  • Style? — clean/minimal, technical/blueprint, whiteboard sketch, polished/presentation-ready
  • Aspect ratio? — 1:1, 16:9, 3:2, 9:16
  • How many variations? — 1-4 (each a separate API call with the same prompt)
  • Where to save? — current directory, ./images/, ~/Desktop, or custom path

Models

Pro

gemini-3-pro-image-preview

  • Highest quality
  • Best for detailed diagrams
  • Default model
Flash

gemini-3.1-flash-image-preview

  • Faster generation
  • Lower cost
  • Good for quick iterations

Requires GEMINI_API_KEY in .env and pip install google-genai pillow.

Knowledge Layers

Three layers, each serving a different purpose and loading at a different time:

Context Loading Strategy
CLAUDE.md Always loaded

Feature index + mandatory rules + workflow skill references. Survives compaction. Minimal context cost.

Skills (24 auto-loaders) On demand

Cheat sheets with correct syntax, config, and patterns. Loaded when the topic comes up. Prevents hallucinated details. Includes 10 persona prompts.

Research docs (38 folders) When needed

Deep-dive documentation with official excerpts, all config options, and source URLs. Only loaded when explicitly read.

How It Works

1. Always Aware

Every session, Claude sees the feature index and knows what exists. Ask "can you use hooks for this?" — it knows hooks exist and where to find details.

2. Accurate on Demand

Claude knows features from training data but gets specifics wrong. The 17 auto-loader skills inject correct syntax, flags, and patterns when the topic comes up.

3. Deep When Needed

For edge cases, Claude reads the full research.md with official documentation excerpts, all configuration options, and source URLs.

4. Workflow Skills

/serious-init, /serious-conversation, /serious-research, /serious-plan, /serious-code, /serious-review, and /serious-bananas load when invoked — providing step-by-step protocols from ideation through implementation, review, and image generation.

Quick Start

Option A: Setup command Recommended

Install the global skill, then open Claude Code in any project and run:

/serious-init

This copies everything into your current project: CLAUDE.md (appends if one exists), all 24 skills, 38 feature documentation folders, and the v6 implementation plan template.

Variants:

/serious-init --skills-only     # Just skills, no docs or template
/serious-init --docs-only       # Just docs + CLAUDE.md, no skills
/serious-init --no-claude-md    # Skip CLAUDE.md if you have one

Option B: Copy manually

cp /path/to/template/CLAUDE.md ./CLAUDE.md
cp -r /path/to/template/.claude/skills/* .claude/skills/
cp -r "/path/to/template/Claude Code Features" "./Claude Code Features"
cp /path/to/template/_implementation_plan_template_v6.md ./_implementation_plan_template_v6.md

Keeping It Updated

Compiled March 2026. Claude Code evolves rapidly. To refresh:
  1. Open Claude Code in the template directory
  2. Ask Claude to update specific research files with the latest docs
  3. Run /serious-init again in your projects to pick up changes
